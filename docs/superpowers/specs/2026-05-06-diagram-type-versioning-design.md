# Версионирование типов и правил диаграмм

## Контекст

Сейчас диаграмма хранит только `diagram_type_id`. Элементы типа диаграммы, типы связей и матрица правил читаются из текущего состояния этого типа. Из-за этого любое изменение правил сразу влияет на все диаграммы, которые используют этот тип.

Нужно сделать фиксируемую зависимость: при подключении типа диаграммы используется последняя версия, эта версия записывается в диаграмму, а будущие изменения типа не ломают уже созданные диаграммы автоматически.

## Цель

Диаграмма должна быть привязана к конкретной версии типа диаграммы. Если у типа появилась новая версия, пользователь видит предложение обновить зависимость. При обновлении backend проверяет текущую диаграмму против новой версии. Если переход невозможен, текущая версия сохраняется, а пользователь получает список проблем, которые нужно исправить для перехода.

## Что Версионируется

Версионируется весь тип диаграммы целиком:

- базовые свойства, которые влияют на правила, включая `is_free_mode`;
- список типов элементов;
- список типов связей;
- матрица правил соединений.

Имя и описание типа можно менять без новой версии, потому что они не влияют на валидность диаграмм.

## Модель Данных

Добавляется таблица `diagram_type_versions`.

Поля:

- `id UUID PRIMARY KEY`;
- `diagram_type_id UUID NOT NULL REFERENCES diagram_types(id) ON DELETE CASCADE`;
- `version_number INTEGER NOT NULL`;
- `snapshot JSONB NOT NULL`;
- `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`;
- уникальность `(diagram_type_id, version_number)`.

`snapshot` хранит неизменяемый снимок:

```json
{
  "diagram_type": {
    "id": "...",
    "key": "class",
    "is_free_mode": false,
    "metadata": {}
  },
  "element_types": [
    {
      "id": "...",
      "key": "class",
      "name": "Class",
      "shape": "rect",
      "svg_path": null,
      "default_style": {},
      "default_size": { "width": 140, "height": 80 },
      "ports": [],
      "field_schema": []
    }
  ],
  "connection_types": [
    {
      "id": "...",
      "key": "association",
      "name": "Association",
      "color": "#34495e",
      "dash": "",
      "arrow_start": "none",
      "arrow_end": "arrow",
      "directed": true,
      "default_style": {}
    }
  ],
  "rules": [
    {
      "from_element_key": "class",
      "to_element_key": "class",
      "connection_type_key": "association",
      "allowed": true
    }
  ]
}
```

Ключи в правилах фиксируются по `key`, а не по `id`, чтобы версии оставались применимыми после импорта, клонирования и пересоздания строк. `id` сохраняются для UI и обратной совместимости, но проверка правил должна опираться на стабильные ключи.

В `diagram_types` добавляется:

- `current_version_id UUID REFERENCES diagram_type_versions(id)`.

В `diagrams` добавляется:

- `diagram_type_version_id UUID REFERENCES diagram_type_versions(id)`.

Для существующих данных миграция создает версию `1` для каждого типа диаграммы и привязывает все диаграммы к версии `1` своего типа.

## Создание Версий

При создании нового типа диаграммы создается первая версия.

Новая версия создается после изменений, влияющих на валидность:

- создание, изменение или удаление типа элемента;
- создание, изменение или удаление типа связи;
- изменение ячейки матрицы правил;
- массовое изменение правил;
- изменение `is_free_mode`.

Редактирование `name`, `description`, `metadata`, если оно не влияет на правила, не создает новую версию.

Версия создается после успешной записи изменений в текущие рабочие таблицы. `diagram_types.current_version_id` переключается на созданную версию. Существующие диаграммы остаются на своих `diagram_type_version_id`.

## Проверка Правил

Все проверки соединений должны использовать версию, закрепленную за диаграммой:

- создание соединения;
- изменение типа соединения;
- пересчет `rule_violation`;
- сохранение диаграммы с блоками и связями;
- импорт и замена диаграммы.

Если у диаграммы нет `diagram_type_version_id`, backend должен использовать текущую версию ее `diagram_type_id`, записать ее в диаграмму и дальше работать с ней. Это нужно для обратной совместимости старых данных.

Free mode определяется снимком версии, а не текущим значением в `diagram_types`.

## Обновление Зависимости

Добавляется статус версии диаграммы:

`GET /api/v1/diagrams/:id/type-version-status`

Ответ:

```json
{
  "diagram_type_id": "...",
  "current_version_id": "...",
  "current_version_number": 1,
  "latest_version_id": "...",
  "latest_version_number": 2,
  "has_update": true
}
```

Добавляется попытка обновления:

`POST /api/v1/diagrams/:id/type-version-update`

Успешный ответ:

```json
{
  "success": true,
  "diagram_type_version_id": "...",
  "version_number": 2
}
```

Если переход невозможен, backend возвращает `409 Conflict`:

```json
{
  "error": "Diagram cannot be upgraded to the latest diagram type version",
  "current_version_number": 1,
  "latest_version_number": 2,
  "issues": [
    {
      "kind": "connection_rule_violation",
      "connection_id": "...",
      "from_block_id": "...",
      "to_block_id": "...",
      "from_element_type": "class",
      "to_element_type": "interface",
      "connection_type": "inheritance",
      "message": "Связь inheritance между class и interface запрещена в версии 2"
    }
  ]
}
```

Возможные `kind`:

- `unknown_element_type`: блок использует тип элемента, которого нет в новой версии;
- `unknown_connection_type`: связь использует тип связи, которого нет в новой версии;
- `connection_rule_violation`: новая версия запрещает существующую связь.

Обновление выполняется в транзакции. Если есть хотя бы одна проблема, `diagrams.diagram_type_version_id` не меняется, `rule_violation` не пересчитывается по новой версии, history snapshot не создается.

Если проблем нет, backend:

1. записывает `diagrams.diagram_type_version_id = latest_version_id`;
2. пересчитывает `rule_violation` по новой версии;
3. записывает snapshot истории;
4. возвращает успешный ответ.

## Frontend

При загрузке диаграммы frontend запрашивает статус версии. Если `has_update = true`, показывает заметное уведомление в рабочей области: "Доступна новая версия правил" и кнопку "Обновить".

После нажатия "Обновить":

- при успехе frontend перезагружает состояние диаграммы, каталог типа и историю;
- при `409` показывает список проблем и оставляет текущую версию без визуальной подмены правил.

Текст ошибок должен говорить пользователю, что именно нужно поправить: заменить неизвестный тип элемента, заменить неизвестный тип связи или удалить/заменить запрещенную связь.

## Импорт И Экспорт

Экспорт диаграммы должен включать версию типа, на которой сейчас зафиксирована диаграмма. Если версии нет из-за старых данных, экспорт использует текущую версию типа после lazy-привязки.

Импорт кастомного типа создает новый тип диаграммы и первую версию из импортированного bundle. Импорт встроенного типа привязывается к текущей версии встроенного типа.

## История

Snapshot истории должен включать `diagram_type_version_id` в объекте `diagram`. При undo/redo восстановление должно возвращать не только `diagram_type_id`, но и версию типа. Это сохраняет корректность старых состояний после обновления правил.

## Границы

Автоматическая миграция элементов и связей между версиями не входит в задачу. Если новая версия удаляет или переименовывает тип, пользователь получает ошибку и сам исправляет диаграмму.

Сравнение версий по diff не требуется. Для MVP достаточно знать текущий номер версии диаграммы и последний номер версии типа.

Совместное редактирование и push-уведомления не входят в задачу. Статус новой версии проверяется при загрузке диаграммы и после ручного обновления каталога.

## Тестирование

Backend:

- миграция создает версии для существующих типов;
- новая диаграмма получает последнюю версию типа;
- изменение правил создает новую версию, но существующая диаграмма остается на старой;
- создание соединения проверяется по версии диаграммы, а не по последней версии типа;
- успешное обновление зависимости меняет версию и пишет history snapshot;
- неуспешное обновление возвращает `409` с issues и не меняет версию;
- undo/redo восстанавливает `diagram_type_version_id`.

Frontend:

- сервисы умеют получать статус версии и запускать обновление;
- при доступном обновлении показывается баннер;
- при ошибке обновления показывается список проблем;
- при успехе состояние диаграммы перезагружается.
